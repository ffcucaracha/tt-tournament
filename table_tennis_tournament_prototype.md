# Прототип приложения для учёта турнира по настольному теннису

## 1. Назначение системы

Приложение предназначено для организации и сопровождения любительского турнира по настольному теннису внутри сообщества, где участники предварительно регистрируются через Google Forms, Яндекс Формы или аналогичный опросник.

Система должна позволять:

- импортировать или вручную заводить участников турнира;
- хранить данные участников, включая ник, трайб и контакт в Telegram;
- случайным образом выполнять посев участников;
- строить турнирную сетку по двойной олимпийской системе;
- отображать список участников, расписание игр, сетку и результаты;
- фиксировать результаты матчей;
- рассчитывать итоговые места всех участников;
- начислять баллы участникам по итоговому месту;
- суммировать и усреднять баллы по трайбам;
- показывать публичные экраны для участников и зрителей;
- поддерживать режим автоматического перелистывания экранов для вывода на общий монитор;
- в перспективе отправлять уведомления в Telegram-канал или чат о ближайших играх.

## 2. Основной сценарий использования

1. В Telegram-группе публикуется ссылка на Google Forms, Яндекс Формы или другой опросник.
2. Участник заполняет форму:
   - ник участника;
   - трайб: «Комета», «Спутник» или «Звезда»;
   - Telegram-ник или номер для связи.
3. Ответы из формы попадают в Google Sheets / Яндекс Таблицу / CSV-файл.
4. Администратор импортирует участников в систему автоматически или вручную.
5. В системе формируется список зарегистрированных участников.
6. Перед ником каждого участника отображается аватар/иконка его трайба.
7. Администратор нажимает кнопку «Посеять».
8. Система случайным образом присваивает каждому участнику уникальный номер посева от `1` до `N`, где `N` — количество зарегистрированных участников.
9. На основании номеров посева система строит сетку турнира по двойной олимпийской системе.
10. Администратор фиксирует результаты матчей.
11. Система обновляет сетку, расписание ближайших игр и итоговую таблицу.
12. После завершения турнира система рассчитывает:
    - место каждого участника;
    - баллы каждого участника;
    - сумму баллов по каждому трайбу;
    - средний балл по каждому трайбу.

## 3. Роли пользователей

### 3.1. Вариант A: только администратор, остальные без авторизации

Минимальный и наиболее простой вариант для прототипа.

Роли:

| Роль | Описание |
|---|---|
| Администратор | Управляет турниром, импортирует участников, выполняет посев, вносит результаты матчей |
| Публичный пользователь | Без авторизации просматривает список участников, расписание, сетку и результаты |

Плюсы:

- проще реализовать;
- не нужно делать личные кабинеты участников;
- достаточно одной административной учётной записи;
- удобно для турнира на один вечер.

Минусы:

- участник не может сам подтвердить участие, изменить контакт или внести результат;
- все действия выполняет администратор;
- меньше контроля над ошибками со стороны участников.

Рекомендуется для MVP.

Участник не является пользователем системы. Это просто сущность в базе данных.

Хранится:

- ник;
- трайб;
- Telegram-контакт;
- номер посева;
- статус участия;
- итоговое место;
- итоговые баллы.

Все действия с участниками выполняет администратор.

Этот вариант оптимален для быстрого запуска.

## 4. Основные сущности

### 4.1. Турнир

Турнир — основная сущность, в рамках которой хранятся участники, матчи, сетка и результаты.

Поля:

| Поле | Тип | Описание |
|---|---|---|
| id | UUID / integer | Идентификатор турнира |
| title | string | Название турнира |
| description | text | Описание турнира |
| status | enum | Статус турнира |
| format | enum | Формат турнира, для MVP — double_elimination |
| sets_to_win | integer | Количество выигранных партий для победы в матче, по умолчанию `2` |
| created_at | datetime | Дата создания |
| started_at | datetime/null | Дата начала |
| finished_at | datetime/null | Дата завершения |

Статусы турнира:

| Статус | Описание |
|---|---|
| pending | Подготовка: загрузка участников и подготовка к посеву |
| in_progress | Турнир идёт |
| finished | Турнир завершён |

### 4.2. Участник

Поля:

| Поле | Тип | Описание |
|---|---|---|
| id | UUID / integer | Идентификатор участника |
| tournament_id | FK | Турнир |
| nickname | string | Ник участника |
| tribe | enum | Комета / Спутник / Звезда |
| telegram_contact | string/null | Telegram-ник или номер для связи |
| registration_order | integer | Порядок регистрации/import-а |
| seed_number | integer/null | Номер посева |
| status | enum | Статус участия |
| final_place | integer/null | Итоговое место |
| final_score | integer/null | Итоговые баллы |
| created_at | datetime | Дата добавления |
| updated_at | datetime | Дата обновления |

Статусы участника:

| Статус | Описание |
|---|---|
| registered | Зарегистрирован |
| confirmed | Подтверждён администратором |
| declined | Отказался / удалён из турнира |
| eliminated | Выбыл из турнирной борьбы |
| finished | Завершил турнир |


### 4.3. Трайб

Для MVP трайбы можно хранить как enum.

Значения:

| Код | Название | Отображение |
|---|---|---|
| comet | Комета | Аватар/иконка кометы |
| satellite | Спутник | Аватар/иконка спутника |
| star | Звезда | Аватар/иконка звезды |


### 4.4. Матч

Поля:

| Поле | Тип | Описание |
|---|---|---|
| id | UUID / integer | Идентификатор матча |
| tournament_id | FK | Турнир |
| bracket_type | enum | Верхняя сетка / нижняя сетка / финал |
| round_number | integer | Номер раунда |
| match_number | integer | Номер матча в раунде |
| participant_a_id | FK/null | Первый участник |
| participant_b_id | FK/null | Второй участник |
| winner_id | FK/null | Победитель |
| loser_id | FK/null | Проигравший |
| score_a | integer/null | Выигранные партии первого участника |
| score_b | integer/null | Выигранные партии второго участника |
| status | enum | Статус матча |
| scheduled_at | datetime/null | Согласованные дата и время матча |
| schedule_source | enum/null | Способ назначения времени: manual / auto |
| created_at | datetime | Дата создания |
| updated_at | datetime | Дата обновления |

Статусы матча:

| Статус | Описание |
|---|---|
| waiting_for_participants | Матч ожидает формирования пары |
| pending | В ожидании: оба участника известны, они согласовывают дату и время |
| in_progress | Дата и время согласованы, матч можно играть |
| finished | Результат внесён |
| cancelled | Матч отменён |

Когда оба участника матча становятся известны, матч автоматически переходит в `pending` и появляется в расписании как «В ожидании».

Переход в `in_progress` выполняет администратор после того, как участники договорились о дате и времени матча.

### 4.5. Результат матча

Для MVP результат можно хранить прямо в таблице `matches`.

Матч игрока A против игрока B ведётся до заданного количества побед по партиям.

Количество побед задаётся настройкой турнира `sets_to_win`. Для первого прототипа значение по умолчанию — `2`, то есть матч играется до двух побед.

Если нужна детализация по партиям, можно добавить отдельную сущность `match_sets`.

Поля `match_sets`:

| Поле | Тип | Описание |
|---|---|---|
| id | UUID / integer | Идентификатор партии |
| match_id | FK | Матч |
| set_number | integer | Номер партии |
| score_a | integer | Очки первого участника |
| score_b | integer | Очки второго участника |

Для простого турнира достаточно хранить только итог по партиям, например `2:0` или `2:1` для матча до двух побед.

### 4.6. Итоговая таблица

Итоговая таблица может рассчитываться динамически или храниться после завершения турнира.

Поля:

| Поле | Тип | Описание |
|---|---|---|
| participant_id | FK | Участник |
| final_place | integer | Итоговое место |
| final_score | integer | Баллы |
| tribe | enum | Трайб |

Правило начисления баллов:

Если в турнире `N` участников, то:

- 1 место получает `N` баллов;
- 2 место получает `N - 1` балл;
- 3 место получает `N - 2` балла;
- последнее место получает `1` балл.

Формула:

```text
score = total_participants - final_place + 1
```

Пример для 20 участников:

| Место | Баллы |
|---:|---:|
| 1 | 20 |
| 2 | 19 |
| 3 | 18 |
| ... | ... |
| 20 | 1 |

### 4.7. Статистика по трайбам

Для каждого трайба рассчитывается:

| Показатель | Описание |
|---|---|
| participants_count | Количество участников от трайба |
| total_score | Сумма баллов участников трайба |
| average_score | Средний балл по трайбу |
| best_place | Лучшее место участника трайба |
| worst_place | Худшее место участника трайба |

Формула среднего балла:

```text
average_score = total_score / participants_count
```

Если в трайбе нет участников, средний балл не рассчитывается или отображается как `0` / `—`.

### 4.8. Журнал аудита

Журнал аудита хранит важные административные действия, чтобы можно было восстановить, кто и когда изменил критичные данные турнира.

Для MVP обязательно логировать:

- посев участников;
- повторный посев;
- внесение результата матча;
- исправление результата матча;
- сброс результата матча.

Поля:

| Поле | Тип | Описание |
|---|---|---|
| id | UUID / integer | Идентификатор записи аудита |
| tournament_id | FK | Турнир |
| admin_id | FK/null | Администратор, выполнивший действие |
| action | enum | Тип действия |
| entity_type | enum | Тип сущности: tournament / match / participant |
| entity_id | UUID / integer/null | Идентификатор изменённой сущности |
| before_json | json/null | Состояние до изменения |
| after_json | json/null | Состояние после изменения |
| metadata_json | json/null | Дополнительные данные действия |
| created_at | datetime | Дата и время действия |

Примеры `action`:

| Код | Описание |
|---|---|
| tournament.seed | Первичный посев участников |
| tournament.reseed | Повторный посев участников |
| match.result.create | Первое внесение результата матча |
| match.result.update | Исправление результата матча |
| match.result.reset | Сброс результата матча |

Для посева в `after_json` нужно хранить снимок распределения участников по номерам посева.

Для изменения результата матча в `before_json` и `after_json` нужно хранить победителя, счёт, статус матча и связанные поля продвижения по сетке, если они изменились.

## 5. Импорт участников

### 5.1. Источник данных

Возможные источники:

1. Google Forms → Google Sheets.
2. Яндекс Формы → Яндекс Таблица / CSV.
3. Ручной импорт CSV.
4. Ручное добавление участников через админку.
5. В перспективе — Telegram-бот.

### 5.2. Минимальный набор полей формы

| Поле формы | Обязательность | Описание |
|---|---:|---|
| Ник | Да | Отображаемое имя участника |
| Трайб | Да | Один из вариантов: Комета, Спутник, Звезда |
| Telegram | Желательно | Ник или номер для связи |

### 5.3. Импорт через CSV

Для MVP рекомендуется начать с CSV-импорта.

Формат CSV:

```csv
nickname,tribe,telegram_contact
player_1,comet,@player_1
player_2,star,@player_2
player_3,satellite,@player_3
```

Плюсы:

- не нужно сразу интегрироваться с Google API или Яндекс API;
- администратор может скачать ответы формы в CSV;
- проще отлаживать;
- меньше внешних зависимостей.

Минусы:

- импорт не полностью автоматический;
- нужно следить за форматом файла.

### 5.4. Автоматический импорт из Google Sheets

Вариант для развития.

Возможные способы:

1. Google Sheets API.
2. Публичная ссылка на CSV-экспорт таблицы.
3. Apps Script, который отправляет webhook в приложение при новой заявке.

Рекомендуемый путь:

- для MVP — CSV;
- для версии 2 — Google Sheets API или webhook.

### 5.5. Правила валидации при импорте

| Проверка | Правило |
|---|---|
| Ник обязателен | Пустые ники не импортируются |
| Трайб обязателен | Значение должно быть одним из допустимых |
| Дубликаты ников | Должны подсвечиваться администратору |
| Telegram-контакт | Может быть пустым, но желательно предупреждение |
| Максимум участников | Можно ограничить настройкой турнира |

### 5.6. Обработка дублей

Запретить импорт дублей по нику.

- ник внутри одного турнира должен быть уникальным;
- при дубле строка не импортируется и отображается в отчёте импорта.

## 6. Посев участников

### 6.1. Назначение посева

Посев нужен, чтобы случайным образом определить положение участников в турнирной сетке.

После посева каждому участнику присваивается уникальный номер от `1` до `N`.

### 6.2. Кнопка «Посеять»

Кнопка доступна администратору, если:

- турнир находится в статусе `pending`;
- в турнире есть минимум 2 участника;
- администратор подтвердил запуск посева.

### 6.3. Правила посева

1. Система берёт всех активных участников турнира.
2. Перемешивает список случайным образом.
3. Присваивает номера посева от `1` до `N`.
4. Сохраняет номера в карточках участников.
5. Меняет статус турнира на `in_progress` автоматически.
6. Строит турнирную сетку.
7. Записывает действие в журнал аудита с итоговым распределением посева.

### 6.4. Повторный посев

В статусе `in_progress` посев недоступен.

Для возврата к подготовке администратор может вручную перевести турнир в `pending` только с подтверждением.

При таком переходе выполняется soft-delete всех результатов турнира:

```text
очищаются winner/loser/score у матчей,
очищаются итоговые места и баллы участников.
```

После возврата в `pending` администратор может выполнить посев заново.

## 7. Турнирная система

### 7.1. Формат: двойная олимпийская система

Двойная олимпийская система означает, что участник выбывает из борьбы за первое место после второго поражения.

Есть две основные сетки:

1. Верхняя сетка — winners bracket.
2. Нижняя сетка — losers bracket.

Участник начинает в верхней сетке.

Если участник проигрывает в верхней сетке, он переходит в нижнюю сетку.

Если участник проигрывает в нижней сетке, он выбывает из турнира.

Победитель верхней сетки и победитель нижней сетки встречаются в финале.

### 7.2. Особенности для количества участников не равного степени двойки

Если количество участников не является степенью двойки, сетка строится с `bye`-слотами.

Пример:

- зарегистрировано 20 участников;
- ближайшая степень двойки — 32;
- в сетке будет 32 стартовых слота;
- 12 слотов будут `bye`;
- часть участников автоматически проходит в следующий раунд.

### 7.3. Варианты реализации сетки

#### Вариант A: собственная реализация генератора сетки

Плюсы:

- полный контроль над логикой;
- можно адаптировать под локальные правила;
- не зависит от внешних библиотек.

Минусы:

- двойная олимпийская сетка сложнее, чем кажется;
- нужно тщательно тестировать переходы из верхней сетки в нижнюю;
- сложно корректно обрабатывать `bye`, финал и сброс финала.

#### Вариант B: использовать готовую библиотеку для bracket-management

Плюсы:

- меньше риска ошибок в турнирной логике;
- готовые модели раундов, матчей и переходов;
- проще поддерживать разные форматы турниров.

Минусы:

- нужно адаптировать структуру БД под библиотеку;
- возможны ограничения кастомизации;
- нужно проверить совместимость с выбранным стеком.

Для MVP рекомендуется рассмотреть готовую библиотеку для управления сетками, а собственную логику оставить только для расчёта баллов, статистики и отображения.


## 8. Расписание игр

### 8.1. Назначение

Расписание игр нужно, чтобы участники и организаторы понимали, какие пары уже сформированы и какие матчи готовы к проведению.

Отдельная публичная страница должна показывать ближайшие игры.

Отдельная сущность для игровых столов не создаётся: для прототипа считается, что доступен один игровой стол.

### 8.2. Ближайшие игры

В расписание попадает матч, у которого известны оба участника.

После появления матча в расписании участникам даётся время договориться о дне и времени игры вне системы, например в Telegram.

Сценарий согласования:

1. Пара участников стала известна.
2. Матч автоматически получает статус `pending` и отображается как «В ожидании».
3. Участники договариваются о дне и времени матча.
4. Участники сообщают договорённость администратору.
5. Администратор вносит дату и время в карточку матча.
6. Матч переходит в статус `in_progress`.

Если участники не договорились, администратор нажимает кнопку «Автоматически».

В этом случае система заполняет `scheduled_at` ближайшей будущей пятницей в `18:30` и переводит матч в статус `in_progress`. Если сейчас уже пятница после `18:30`, выбирается следующая пятница.

На публичном экране отображаются:

- ближайшие 5 матчей со статусом `in_progress`;
- ближайшие 5 матчей со статусом `pending`.

Для каждого матча показывается:

- номер матча;
- раунд;
- тип сетки: верхняя / нижняя / финал;
- ник первого участника;
- ник второго участника;
- аватарчики трайбов;
- согласованные дата и время, если они уже внесены;
- текущий статус.

Пример:

| Статус | Матч | Сетка | Игрок 1 | Игрок 2 | Время |
|---|---|---|---|---|---|
| Согласовано | M12 | Верхняя, раунд 2 | ☄️ alex | ⭐ maria | Пт 18:30 |
| В ожидании | M13 | Верхняя, раунд 2 | 🛰️ nick | ☄️ den | Согласовывается |
| Согласовано | M14 | Нижняя, раунд 1 | ⭐ anna | 🛰️ kirill | Сб 12:00 |

### 8.3. Доступность матчей

Очерёдность матчей не важна и не хранится отдельным полем.

Главное условие доступности матча — известная пара участников.

Правила:

- если один или оба участника ещё неизвестны, матч находится в `waiting_for_participants`;
- если оба участника известны, матч переходит в `pending`;
- если дата и время согласованы, матч переходит в `in_progress`;
- матч в `pending` или `in_progress` не блокирует другие матчи;
- зависимые матчи сетки ожидают только результатов предыдущих матчей, из которых должны прийти участники.

Сортировка на публичном экране нужна только для удобства отображения:

- сначала `in_progress` по ближайшему `scheduled_at`;
- затем `pending` по `round_number` и `match_number`.

## 10. Режим автоматического показа на экране

### 10.1. Назначение

Режим нужен для вывода сайта на общий экран во время турнира.

Публичная страница должна автоматически переключать информационные экраны по кругу.

### 10.2. Состав экранов

Рекомендуемый цикл:

1. Участники с номерами посева.
2. Расписание ближайших игр.
3. Турнирная сетка.
4. Итоговые результаты / статистика по трайбам.

Если нужно строго три экрана, можно объединить итоговые результаты и статистику по трайбам в один экран.

### 10.3. Поведение

- Каждые 10 секунд система переключает экран на следующий.
- После последнего экрана цикл начинается заново.
- Должна быть возможность открыть каждый экран отдельно.
- Должна быть отдельная ссылка на режим слайд-шоу, например `/display`.
- Желательно добавить полноэкранный режим в браузере.

### 10.4. Параметры URL

Можно добавить параметры:

```text
/display?interval=10
/display?interval=15&screens=participants,schedule,bracket,tribes
```

Для MVP достаточно фиксированного интервала 10 секунд.

## 11. Административные страницы

### 11.1. Авторизация администратора

Для MVP достаточно простой авторизации администратора:

- email + пароль;
- одна или несколько учётных записей;
- закрытый административный раздел `/admin`.

### 11.2. Админка турниров

Функции:

- создать турнир;
- редактировать название и описание;
- настроить количество побед по партиям для победы в матче;
- выполнить посев;
- просмотреть запись аудита по посеву;
- вернуть турнир из `in_progress` в `pending` с подтверждением и soft-delete результатов;
- завершить турнир;
- просмотреть текущий статус турнира.

### 11.3. Админка участников

Функции:

- просмотреть список участников;
- добавить участника вручную;
- отредактировать участника;
- удалить участника до старта турнира;
- импортировать CSV;
- увидеть ошибки импорта;
- выполнить сортировку по регистрации, нику, трайбу, номеру посева.

### 11.4. Админка матчей

Функции:

- просмотреть матчи по сеткам и раундам;
- увидеть матчи в ожидании и готовые матчи;
- внести согласованные дату и время матча;
- автоматически назначить ближайшую пятницу в 18:30;
- внести результат;
- исправить результат, если следующий зависимый матч ещё не сыгран;
- просмотреть историю внесения и изменения результата;
- отметить матч как начатый;
- отметить матч как завершённый.

### 11.5. Ввод результата матча

Минимальная форма:

| Поле | Описание |
|---|---|
| Победитель | Выбор одного из двух участников |
| Счёт | Например `2:0` или `2:1` для матча до двух побед |

Расширенная форма:

| Поле | Описание |
|---|---|
| Партия 1 | Счёт по очкам |
| Партия 2 | Счёт по очкам |
| Партия 3 | Счёт по очкам, если нужна |
| Победитель | Определяется автоматически |

Для MVP достаточно итогового счёта по партиям. Необходимое количество побед берётся из настройки турнира `sets_to_win`.

Каждое внесение, исправление или сброс результата матча должно создавать запись в журнале аудита.

### 11.6. Админская страница аудита

В административном разделе должна быть отдельная страница просмотра аудита, например `/admin/audit`.

Страница доступна только администратору.

Назначение страницы — показать журнал критичных действий по турниру и дать быстро проверить, кто и когда выполнил посев или изменил результат матча.

Функции:

- список записей аудита в обратном хронологическом порядке;
- фильтр по типу действия: посев, повторный посев, внесение результата, исправление результата, сброс результата;
- фильтр по матчу или участнику, если действие связано с конкретной сущностью;
- фильтр по администратору;
- фильтр по периоду;
- отображение администратора, даты и времени действия;
- просмотр краткого описания действия;
- просмотр состояния до и после изменения в детальной карточке или модальном окне;
- просмотр снимка распределения посева.

Минимальные колонки таблицы:

| Колонка | Описание |
|---|---|
| Дата и время | Когда выполнено действие |
| Администратор | Кто выполнил действие |
| Действие | Тип действия |
| Сущность | Турнир, матч или участник |
| Описание | Краткое человекочитаемое описание изменения |

Для записей по результатам матча описание должно показывать пару участников и счёт до/после.

Для записей по посеву описание должно показывать количество участников и ссылку на просмотр снимка распределения посева.

## 12. Расчёт итоговых мест

### 12.1. Проблема

В double elimination не всегда очевидно, как присвоить уникальные места всем участникам, особенно если несколько участников выбывают на одном этапе.

По требованию системы каждый участник должен получить отдельное место: от `1` до `N`.

### 12.2. Определения мест

1. Первое и второе место определяются финалом.
2. Остальные места определяются по порядку выбывания.
3. При одинаковом этапе выбывания применяются tie-breaker правила:
   - количество побед;
   - разница выигранных и проигранных партий;

## 15. Нефункциональные требования

### 15.1. Производительность

Приложение рассчитано на малую нагрузку:

- десятки участников;
- десятки или сотни зрителей публичной страницы;
- один или несколько администраторов.

### 15.2. Надёжность

Важно:

- не терять результаты матчей;
- запрещать случайное пересоздание сетки после старта;
- логировать административные действия по посеву и результатам матчей;
- хранить состояние до и после критичных изменений;
- иметь резервную копию базы данных перед турниром и после турнира.

### 15.3. Безопасность

Требования:

- административный раздел закрыт авторизацией;
- публичные страницы доступны только на чтение;
- секреты хранятся в `.env`;
- Telegram Bot Token не попадает в репозиторий;
- импорт CSV валидируется;
- ввод результата проверяется на корректность;
- нельзя изменить результат матча, если зависимые матчи уже сыграны, без специального сброса.

### 15.4. UX

Требования:

- крупный читаемый интерфейс для вывода на экран;
- минимум мелких элементов на публичных страницах;
- визуальные иконки трайбов;
- понятная подсветка готовых матчей;
- быстрая фиксация результата администратором;
- адаптивность под ноутбук, планшет и большой экран.

## 18. Docker и окружения

### 18.1. Контейнеры

Рекомендуемый состав:

```text

```

Для разработки можно запускать:

```text
frontend-dev
```

Особенности:

- frontend запускается через Vite dev server;
- backend запускается в watch-режиме;
- код монтируется через volumes;
- база данных запускается в контейнере;
- миграции можно запускать отдельной командой.

Пример команд:

```bash
docker compose -f docker-compose.dev.yml up --build
```

Особенности:

- frontend собирается в статические файлы;
- backend запускается как Node.js-сервис;
- nginx отдаёт frontend и проксирует API;
- база PostgreSQL хранит данные в volume;
- секреты берутся из `.env` на сервере.

Пример команд:

```bash
docker compose -f docker-compose.prod.yml up -d --build
```

### 18.4. Переменные окружения

Пример `.env`:

```env

PUBLIC_APP_URL=https://example.com
```

Секреты не должны попадать в git.

4. Создать `.env` на сервере.
5. Запустить production compose.
6. Настроить домен и HTTPS через nginx/Traefik/Caddy.

Команды общего вида:

```bash
git clone <repo-url>
cd table-tennis-tournament
cp .env.example .env
nano .env
docker compose -f docker-compose.prod.yml up -d --build
```

### 19.2. Более быстрый деплой через скрипт

Можно добавить скрипт `deploy.sh`:

```bash
#!/usr/bin/env bash
set -e

git pull origin main
docker compose -f docker-compose.prod.yml build
docker compose -f docker-compose.prod.yml up -d
docker compose -f docker-compose.prod.yml exec backend npx prisma migrate deploy
```

### 19.3. Деплой через GitHub Actions / GitLab CI

Более продвинутый вариант:

1. При push в `main` запускается pipeline.
2. Собираются Docker-образы.
3. Образы публикуются в registry.
4. Сервер подтягивает свежие образы.
5. Выполняются миграции.
6. Контейнеры перезапускаются.

Плюсы:

- меньше ручных действий;
- воспроизводимая доставка;
- удобно для дальнейшей поддержки.

Минусы:

- нужно настраивать registry, SSH-ключи и секреты CI.

### 19.4. Рекомендация по доставке

Для первого запуска:

- VPS + Docker Compose + ручной `git pull` + `docker compose up -d --build`.

Для дальнейшего развития:

- GitLab CI / GitHub Actions + Docker Registry + автоматический деплой по SSH.

## 22. Риски и спорные места

### 22.1. Double elimination сложнее простой сетки

Риск: ошибки в переходах между верхней и нижней сеткой.

Меры:

- использовать готовую библиотеку или тщательно покрыть генератор тестами;
- отдельно протестировать количество участников 3, 4, 5, 7, 8, 10, 16, 20;
- проверить сценарии с `bye`.

### 22.2. Уникальные места для всех участников

Риск: классическая сетка может давать группы участников с одинаковым условным местом.

Меры:

- заранее зафиксировать правила tie-breaker;
- добавить ручную корректировку итоговых мест администратором.

### 22.3. Импорт из внешних форм

Риск: Google/Яндекс API могут потребовать отдельной настройки доступа.

Меры:

- для MVP использовать CSV;
- автоматический импорт вынести во вторую версию.

### 22.4. Telegram-уведомления

Риск: возможны лишние или повторные сообщения.

Меры:

- для MVP сделать ручную отправку;
- хранить историю отправленных уведомлений;
- не делать автоматическую отправку без антиспама.

## 23. Открытые вопросы

1. Нужен ли grand final reset или финал играется один раз? - один раз
2. Нужно ли хранить счёт по каждой партии или только итог `2:0` / `2:1`? - только итог
3. Нужно ли участникам самим подтверждать результаты? - нет, они пишут в лс админу
4. Нужно ли делать несколько турниров в одной системе или только один активный турнир? - только один активный турнир
5. Как именно распределять уникальные места среди участников, выбывших на одном этапе?
6. Должна ли статистика по трайбам считаться только по среднему баллу или ещё по сумме? - только по среднему баллу
7. Нужно ли скрывать Telegram-контакты на публичных страницах? - да
8. Нужна ли поддержка редактирования результата после внесения? - да
